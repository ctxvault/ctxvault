# Projection Rights: Why Candidate Context Needs Evidence Before It Reaches AI Tools

Date: 2026-05-08
Status: v0.4.1 design preview.

This is not a v0.4.0 shipped-feature claim.

| Boundary | Status |
| --- | --- |
| v0.4.0 shipped behavior | Local sources -> reviewed context -> receipt-backed handoff |
| v0.4.1 design target | Projection eligibility kernel |
| Stable external API | Not shipped |
| External memory adapters | Not shipped |
| Runtime control over AI tools | Not shipped |
| Hallucination-prevention claim | Not claimed |
| Memory benchmark claim | Not claimed |

## Purpose

When an AI tool sees old context, the hard question is not whether that context
was stored. The hard question is whether it is allowed to influence the next
run.

CtxVault's v0.4.1 schema work is about that permission boundary. The project
should not ask "what can the system remember?" first. It should ask which past
information is qualified to influence the next AI work surface, with what
source evidence, scope, review state, policy, and receipt.

That means raw notes, transcripts, summaries, retrieval results, verifier
outputs, and external memory-engine reports are candidate material until a
projection gate authorizes a route.

## Core Rule

```text
candidate material is not context until projection is authorized
```

The v0.4.1 design keeps source support, policy, review, and final route
ownership separate:

```text
SourceRef
  -> MemoryCandidate / ClaimRecord / context candidate
  -> EvidenceBinding
  -> ClaimSupportReceipt
  -> PolicyProfile
  -> ProjectionGateDecision / ProjectionGateReceipt
  -> ReceiptBundle
```

`ClaimSupportReceipt` owns support status. `ProjectionGateReceipt` owns the
final route.

## Version Boundary

| Version | What it means | What it does not mean |
| --- | --- | --- |
| v0.4.0 | A deterministic local handoff path: source material becomes reviewed context and receipt-backed projections. | It does not ship projection-rights schemas as product behavior. |
| v0.4.1 | A design and contract lane for deciding whether candidates are eligible to influence AI work surfaces. | It does not ship a stable Candidate/Projection Protocol. |
| v0.5+ | Possible adapter and candidate-provider RFC work after review workflow and gate contracts are concrete. | It does not start in v0.4.1 and must not be implied by v0.4.0 copy. |

## Why This Exists

Most AI-work failures are not only retrieval failures. They are permission
failures:

- unsupported context is presented as fact
- stale project state is reused without a caveat
- retrieved material bypasses review
- model or engine output becomes durable memory too early
- action-adjacent instructions are projected without source and policy checks
- users cannot tell what the AI tool actually saw

Projection rights make those failures inspectable before context reaches the
next work surface.

## Object Roles

### SourceRef

A pointer to the source, revision, span, capture, or artifact that can support a
candidate. A source ref is stronger than a generated summary because it can be
inspected and re-bound.

### MemoryCandidate

Staged durable memory material. It is not `Memory` and not truth. It must
declare purpose, scope, sources, evidence, freshness, and promotion state before
projection.

### MemoryPurposeTaxonomy

The purpose label for a memory candidate. It explains whether the candidate is
context memory, trajectory memory, episodic event memory, semantic fact memory,
preference/profile memory, procedural skill memory, external memory-engine
candidate material, or artifact/consensus memory.

Each purpose declares allowed sources, freshness rules, conflict handling,
projection eligibility, required receipts, and forbidden promotion behavior.

### EvidenceBinding

The source-to-candidate link. It preserves source refs, span refs, excerpt
hashes, relation, and binding method so a reviewer can inspect why a candidate
is considered supported, caveated, contradicted, or unbound.

### ClaimSupportReceipt

The support-status receipt. It records whether a claim is supported,
caveated, contradicted, not in sources, ambiguous, unverifiable by policy,
source-unavailable, verifier-unavailable, blocked by safety policy, or failed
because a receipt could not be written.

It does not decide the final projection route.

### PolicyProfile

The static policy input. It records allowed uses, target risk, action risk,
required review, forbidden runtime behavior, source requirements, and
fail-closed rules.

### ProjectionGateDecision / ProjectionGateReceipt

The final route owner. It decides whether candidate material can be projected,
projected with caveat, held for more source, abstained, sent to manual review,
or blocked.

Allowed routes:

- `allow`
- `allow_with_caveat`
- `needs_more_source`
- `abstain`
- `manual_review`
- `block`

Missing source, contradiction, schema error, verifier failure, receipt write
failure, package identity risk, and command side-effect risk must not silently
allow projection.

### ReceiptBundle

The proof packet. It groups the source, evidence, support, policy, review, and
projection receipts that explain why a handoff was allowed, caveated, withheld,
or blocked.

## Memory Governance Companions

The v0.4.1 memory-governance fixtures add three checks around the taxonomy:

- `DecoupledMemoryEvalMatrix`: separates write, manage, retrieve,
  read/project, update, abstain, conflict, privacy, latency/cost, and review
  burden axes.
- `MemoryMechanismCouplingScorecard`: exposes model, retriever, vector/graph/
  database, benchmark, learned-policy, feedback-loop, hidden-promotion, and
  source-trace coupling.
- `MinimalMemoryBaselineReceipt`: requires exact refs, keyword/BM25, time
  filters, recency/frequency, source trace, explicit abstention, or
  reviewer-selected context before complex mechanisms or benchmark claims.

These are governance receipts. They do not rank memory systems and do not
approve adapters.

## Full Example

Suppose a public draft says:

```text
CtxVault supports external memory adapters.
```

For v0.4.0, that claim must not be projected as a shipped capability:

| Field | Example value |
| --- | --- |
| Candidate | `MemoryCandidate`: "CtxVault supports external memory adapters." |
| Available source | No qualifying v0.4.0 source says this shipped. |
| EvidenceBinding | Not bound. |
| ClaimSupportReceipt | `not_in_sources` |
| PolicyProfile | Stable API and adapter claims are forbidden for v0.4.0 copy. |
| ProjectionGateReceipt | `needs_more_source` or `block` |
| ReceiptBundle | Records why the claim cannot influence the next AI work surface. |

The result is not "the claim is false." The result is narrower and more useful:
CtxVault cannot allow that claim to influence public copy or an AI work packet
without qualified source evidence and the right version scope.

## Boundary

This design preview does not approve:

- external memory adapters
- `EngineProfile` as a v0.4.1 runtime contract
- live retrieval
- live registry or documentation lookup
- provider/model execution
- runtime control over AI tools such as Codex, Claude Code, Cursor, ChatGPT,
  browsers, or terminals
- public benchmark or hallucination-prevention claims
- stable external Candidate/Projection Protocol

Those may be considered later only after the governance kernel and review
workflow can block unsafe projection.
