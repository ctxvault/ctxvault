The v0.4.1 north star remains narrow:

```text

```

routes, fixtures, and tests can be shown; runtime gates, verifiers, Workbench
UX, external adapters, durable memory/consolidation, and anti-hallucination
prevention claims move later. It is not a Memory OS, memory control plane,
unified agent memory platform, stable external API, adapter release, benchmark
claim, public beta readiness claim, scanner product, verifier product, or
runtime control layer.

## Requires Owner Approval

### Option 1: v0.4.1 Release Shape

Choices:

- **A+. Schema Explanation Plus Deterministic Fixture/Test Pack Only.**
  Ship v0.4.1 as docs, schema, receipts, routes, deterministic fixtures,
  focused tests, and public-safe architecture notes. Do not promise runtime
  behavior.
- **B. Schema Plus Runtime Gate Wiring.**
  Add CLI/Workbench behavior that reads the kernel schema and changes real
  projection behavior in user flows.
- **C. Broader Governance Platform Release.**
  Include memory substrate, package-risk, review workflow, and adapter-facing
  language under one v0.4.1 release banner.

Recommendation: **A+**.

Why: v0.4.1's highest leverage is making the object model legible and
verifiable through local fixtures and tests while v0.4.0 feedback is still
fresh. Option B is viable later, but it creates product behavior risk and new
UX obligations. Option C blurs the boundary into Memory OS/control-plane
language and should not be v0.4.1.

### Option 2: Public Timing For v0.4.1 Docs

Choices:

- **A. Publish as design preview in public repo before the v0.4.1 tag.**
  Keep clear copy: design preview, experimental, non-normative, not v0.4.0
  shipped behavior, not stable API.
- **B. Hold all v0.4.1 docs until the v0.4.1 release tag.**
- **C. Announce as a protocol/RFC before release.**

Recommendation: **A**.

Why: A public design preview supports the governance narrative without claiming
release completion. Holding everything reduces learning value. Calling it an
RFC or protocol too early creates ecosystem expectations the repo has not
earned.

### Option 3: Trust-Floor Depth In v0.4.1

Choices:

- **A. Keep the canonical kernel minimal.**
  Source evidence, candidate context, review decisions, projection, handoff
  packet, and receipt states.
- **B. Promote selected trust-floor experimental receipts into canonical v0.4.1
  schema after focused review.**
  Examples: claim support receipts, projection gate reason codes, source
  ingestion gate receipts.
- **C. Add runtime verifier, package resolver, or model-backed checking.**

Recommendation: **A now, B only after a focused approval.**

Why: A keeps the release tight and already gives the project a credible kernel.
B can be high value, but each promoted receipt becomes a compatibility surface
that needs naming discipline. C is explicitly outside v0.4.1 baseline
correctness.

### Option 4: Code And Package Safety Receipts

Choices:

- **A. Keep package/code risk in experimental deterministic fixtures for v0.4.1.**
  Do not promote it to a canonical product surface, and do not run live
  registry or command checks.
- **B. Add canonical deterministic schemas for dependency/API/command receipts,
  but no live registry or execution.**
- **C. Add live registry/API lookups or command execution checks.**

Recommendation: **A for v0.4.1, B for a later approved patch if user feedback
shows code/package trust is the primary blocker.**

Why: Package and command safety is a high-risk area and must keep deterministic
fixture coverage. It should not become the canonical v0.4.1 product surface or
pull CtxVault toward a supply-chain scanner narrative. C would require network,
policy, privacy, caching, and failure-mode design that belongs outside this
release.

### Option 5: Review Workflow / Workbench Integration
