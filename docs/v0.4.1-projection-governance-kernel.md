# CtxVault v0.4.1 Projection Governance Kernel

Date: 2026-05-08
Status: experimental, non-normative v0.4.1 schema explanation and object-model
design preview.

Public-use marker: v0.4.1 object-model document. Use it for the Projection
Governance Kernel explanation, public-safe boundaries, and schema non-goals.

This is not a v0.4.0 shipped-feature claim. It is not a standard protocol
release, not a stable external API, not runtime behavior, and not a
runtime-control contract.

Public hook: Know what your AI tools see.

## Public Boundary

CtxVault v0.4.1 explains the object model behind reviewed context handoff:
sources become candidate context; review decisions govern what may be
projected; receipts record what was selected, omitted, blocked, written, or not
done.

Release posture: schema, receipts, routes, deterministic fixtures, and focused
tests can be shown. Runtime gates, verifiers, Workbench UX, external adapters,
durable memory/consolidation, and anti-hallucination prevention claims move to
later owner-approved work.

The v0.4.0 public release already proves a narrower behavior: local sources can
be turned into reviewed context and receipt-backed handoff artifacts. The
v0.4.1 kernel names the control objects behind that behavior so future work can
stay reviewable and fail closed.

## Non-Goals

The Projection Governance Kernel is deliberately not:

- a Memory OS
- a memory control plane
- a unified agent memory platform
- an external stable Candidate/Projection Protocol
- a Mem0, Zep, Graphiti, Cognee, Redis, or vector-database adapter layer
- a benchmark paper or public ranking against memory systems
- a public beta readiness claim
- a v0.5 Candidate/Projection Protocol RFC
- a claim to control Codex, Claude Code, Cursor, ChatGPT, or any AI-tool runtime

Those terms can appear in roadmap discussions only as excluded or deferred
scope. They must not become v0.4.0 or v0.4.1 shipped claims.

## Object Model

```text
Source Evidence
  -> Candidate Context
  -> Review Decision
  -> Projection
  -> Receipt
  -> Handoff Packet / Work Surface
```

### Source Evidence

Source evidence is the raw material a reviewer can inspect: files, transcripts,
notes, demo vault entries, release artifacts, or synthetic fixtures. A raw
transcript or source file is evidence, not durable memory and not automatic
truth.

Schema fields:

- `id`
- `kind`
- `path_or_ref`
- optional `digest`
- optional `captured_at`

### Candidate Context

Candidate context is material extracted, retrieved, summarized, or organized
from source evidence. It is useful because it is smaller than raw evidence, but
it is not automatically eligible to influence an AI work surface.

Schema fields:

- `id`
- `source_ids`
- `summary_or_excerpt`
- `scope`
- optional `confidence`
- `status: candidate`

The status is intentionally narrow. A candidate can be useful and still be
omitted or blocked.

### Review Decision

Review decisions convert candidate context into a route decision. The minimal
state set is:

- `selected`
- `omitted`
- `blocked`

Each decision carries a `reason`, `scope`, and optional reviewer/timestamp. The
review decision owns whether a candidate is selected, omitted, or blocked. It
does not write the final handoff artifact by itself.

### Projection

Projection is the control point. It says which selected candidates may be
included in the context package for a target work surface. Projection is where
old information becomes eligible to influence the next AI work step.

Schema fields:

- `id`
- `target_surface`
- `included_candidate_ids`
- `scope`
- `created_at`

The target surface is an artifact surface such as `AGENTS.md`, `CLAUDE.md`, a
workstream brief, ChatGPT Projects material, or a generic handoff packet. It is
not a claim that CtxVault controls the runtime behind that surface.

### Receipt

Receipts record what happened in the handoff. The minimal v0.4.1 receipt state
set is:

- `selected`
- `omitted`
- `blocked`
- `written`
- `not_done`

`selected`, `omitted`, and `blocked` refer to candidate decisions. `written`
records artifacts that were written or handed off. `not_done` records explicit
non-actions so roadmap or adapter language does not silently become a shipped
claim.

### Handoff Packet / Work Surface

A handoff packet is the artifact that crosses into a work surface. The v0.4.1
schema records a packet reference and a target surface, plus
`runtime_control_claim: not_claimed`. That field is intentionally explicit:
CtxVault can prepare governed context artifacts without claiming to control
Codex, Claude Code, Cursor, ChatGPT, or another runtime.
